1. Java 17 & OOP

OOP — 4 Pillars at Apply Level
Encapsulation — hide state behind accessors; prevents invariant violations. Not just private + getters; it's about protecting consistency.
Inheritance — "is-a" relationship; use sparingly. Deep hierarchies become fragile (superclass changes break all subclasses).
Polymorphism — one interface, many implementations. Runtime dispatch via vtable. Enables open/closed principle.
Abstraction — expose what's necessary, hide the rest. Interface = contract; abstract class = partial implementation + shared state.
"Abstract class vs interface? Interface is a pure contract — no state, default methods aside. Abstract class gives you partial implementation and protected fields. I use interface by default; abstract class when subtypes genuinely share code, not just behaviour."
Abstract class can have constructors — called via super(). Interface cannot have instance state (only static final constants).
Composition > inheritance: Favour has-a — easier to swap behaviour at runtime, no fragile base class problem. 
// Prefer
class OrderService {
    private final PaymentGateway gateway; // injected
}
// Over
class OrderService extends PaymentGateway {}

In Spring, DIP is automatic via DI container. Mention this — it shows you connect principles to the framework.
"The one I see violated most in production is SRP — big service classes that do everything. I aggressively split them even if it means more files."
equals / hashCode Contract + Immutability
Contract: if a.equals(b) then a.hashCode() == b.hashCode(). Converse not required.
If you override one, you must override both — otherwise HashMap breaks (object goes into wrong bucket, never found). 
// Broken: override equals only
map.put(obj, "value");
map.get(obj); // returns null — different hash bucket
Immutability:
  • final class, final fields, no setters, defensive copies in constructor and getters
  • Thread-safe by design — no synchronisation needed
  • String, Integer, LocalDate are immutable; that's why they're safe as HashMap keys
"Why immutability? Three reasons: thread safety for free, safe HashMap keys, and elimination of a whole class of defensive-copy bugs. I default to immutable value objects and only add mutability when there's a real reason."
record gives you immutability and correct equals/hashCode for free — mention this as the modern approach.
Generics — Bounded Wildcards, PECS
PECS — Producer Extends, Consumer Super.
  • ? extends T — you read from it (producer). Can't add (type unknown at runtime).
  • ? super T — you write to it (consumer). Can't read safely beyond Object.
// Producer: copy from source
void copy(List src, List dst) {
    for (Number n : src) dst.add(n);
}
Type erasure: generics are compile-time only. At runtime List<String> and List<Integer> are both List. No instanceof List<String> at runtime.
You cannot create generic arrays: new T[10] won't compile. Workaround: (T[]) new Object[10] with @SuppressWarnings.
In Spring Data / JPA generics, this comes up when creating base repository abstractions. Know that JpaRepository<T, ID> erases at runtime and Spring uses ResolvableType to reconstruct type info.
Collections — Internals That Interviewers Probe
HashMap (Java 8+): array of buckets; each bucket is a linked list, upgraded to red-black tree when bucket size > 8 (treeifies). Load factor default 0.75; resizes (doubles) when exceeded. Key must implement hashCode and equals.
ConcurrentHashMap: segment-level locking (Java 7) replaced by CAS + synchronized on individual bucket head (Java 8). No full-lock; reads are lock-free. Don't use Collections.synchronizedMap in concurrent code.
ArrayListLinkedList
Random accessO(1)O(n)
Insert at endO(1) amortizedO(1)
Insert at middleO(n)O(1) if you have iterator
MemoryCompactNode overhead per element
TreeMap: red-black tree; keys sorted; O(log n) ops. Use when you need floorKey, ceilingKey, range views.
LinkedList is almost never the right answer in practice — cache locality is terrible. Default to ArrayList unless you have a deque use case.
Streams API — map/filter/reduce, Collectors, Parallel Pitfalls
Streams are lazy — intermediate ops (map, filter, flatMap) build a pipeline; nothing runs until a terminal op (collect, forEach, findFirst, reduce). 
List<String> result = orders.stream()
    .filter(o -> o.getStatus() == ACTIVE)
    .map(Order::getCustomerName)
    .distinct()
    .collect(Collectors.toList()); // or Collectors.toUnmodifiableList() in 10+
Key collectors: groupingBy, partitioningBy, joining, toMap, counting, summarizingInt. 
Map<Status, List<Order>> byStatus = orders.stream()
    .collect(Collectors.groupingBy(Order::getStatus));
Parallel streams pitfalls:
  • Uses ForkJoinPool.commonPool — shared with the entire JVM. In a servlet container, this can starve request threads.
  • Not beneficial for small collections or I/O-bound ops.
  • Stateful lambdas + parallel = race conditions.
  • Order not guaranteed unless you use forEachOrdered.
toMap throws IllegalStateException on duplicate keys. Use the 3-arg overload with a merge function.
Optional — Proper Use and Anti-patterns
Optional<T> is a container that may or may not hold a value. Designed for return types only — signals "this might not exist". 
// Good
Optional<User> findById(Long id);

user.findById(id)
    .map(User::getEmail)
    .orElse("unknown");
Anti-patterns:
  • optional.get() without isPresent() — defeats the purpose, throws NoSuchElementException
  • Optional as a method parameter — use overloading instead
  • Optional as a field — not serializable, wasteful
  • Optional.of(null) — throws NPE; use Optional.ofNullable
  • Wrapping collections — return empty list, not Optional<List>
"I use Optional only as a return type to make absence explicit in the API contract. Never as a parameter or field."
orElseGet(() -> expensiveCall()) is lazy; orElse(expensiveCall()) always evaluates the argument — easy perf bug.
Concurrency — synchronized, volatile, ExecutorService, CompletableFuture
synchronized — mutual exclusion + memory visibility. Acquires monitor lock. Can cause deadlock if lock ordering is inconsistent.
volatile — visibility only, no atomicity. Reading/writing a volatile long is atomic, but i++ on volatile is not (read-modify-write is three ops). 
// ExecutorService
ExecutorService pool = Executors.newFixedThreadPool(4);
Future<String> f = pool.submit(() -> fetchData());
String result = f.get(5, TimeUnit.SECONDS);
pool.shutdown();
CompletableFuture: non-blocking async composition. 
CompletableFuture.supplyAsync(() -> fetchUser(id))
    .thenApply(user -> enrich(user))
    .thenAccept(System.out::println)
    .exceptionally(ex -> { log(ex); return null; });
Virtual threads (Java 21 stable, preview in 19-20): in Java 17 they don't exist stably. Say you know they're coming and that they're JVM-managed lightweight threads that unblock the carrier thread on I/O — key for high-throughput services.
Don't mention virtual threads as a Java 17 feature. Stable in 21 only.
Prefer Executors.newVirtualThreadPerTaskExecutor() in Java 21+ over fixed pools for I/O-bound tasks.
Exceptions — Checked vs Unchecked, try-with-resources
Checked (Exception subclasses, not RuntimeException) — compiler forces handling. Appropriate when the caller can reasonably recover (e.g., IOException, SQLException).
Unchecked (RuntimeException and below) — programming errors or unrecoverable states (NullPointerException, IllegalArgumentException). Don't force caller to handle what they can't fix.
"In modern Spring applications I almost exclusively use unchecked exceptions. Checked exceptions in service layers force ugly try-catch boilerplate and often get swallowed. I wrap infrastructure exceptions in unchecked domain exceptions at the boundary."
try-with-resources — any AutoCloseable declared in the try clause is closed automatically, even on exception. Suppressed exceptions are attached to the primary exception. 
try (Connection conn = ds.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // use conn and ps
} // both closed in reverse order
If both the body and close() throw, the close() exception is suppressed — accessible via Throwable.getSuppressed(). The body exception propagates.
Never catch Exception or Throwable silently. And never catch (Exception e) {} — interview red flag.
Functional Interfaces, Lambdas, Method References
A functional interface has exactly one abstract method (SAM). Annotated @FunctionalInterface (optional but documents intent). Can have default/static methods.
InterfaceSignatureUse
Function<T,R>R apply(T t)Transform
Predicate<T>boolean test(T t)Filter
Consumer<T>void accept(T t)Side effect
Supplier<T>T get()Lazy provision
BiFunction<T,U,R>R apply(T,U)Two inputs
Method references — syntactic sugar for lambdas: 
list.forEach(System.out::println);       // instance method of param
list.stream().map(String::toUpperCase);  // instance method
list.stream().map(User::new);            // constructor ref
"A functional interface is just a contract for a single behaviour. Lambdas are anonymous implementations. Method references are shorthand when the lambda body is just a single method call — cleaner and the intent is obvious."
Lambdas capture effectively-final local variables. If you try to mutate a captured variable inside a lambda, it won't compile. Use an array or an AtomicInteger wrapper as a workaround when you must.
Records and Sealed Classes — When to Actually Use Them
Records — transparent immutable data carriers. Use for DTOs, value objects, API request/response bodies, map keys. Not for entities (JPA entities need no-arg constructor and mutable fields). 
record CreateOrderRequest(String customerId, BigDecimal amount) {}
// Compact constructor for validation:
record CreateOrderRequest(String customerId, BigDecimal amount) {
    CreateOrderRequest {
        Objects.requireNonNull(customerId);
        if (amount.signum() <= 0) throw new IllegalArgumentException();
    }
}
Sealed classes — restrict which classes can extend/implement. Compiler can verify exhaustive switch. Use for modelling closed domain hierarchies (payment method types, event types, result types).

2. Spring Core (IoC / DI)

IoC Container — the mental model
Inversion of Control: instead of your code calling new SomeService(), the container creates and wires objects — you declare what you need, Spring decides how to build it.
  • The container owns the object graph. You get a reference, not responsibility for construction.
  • Benefits: testability (swap real deps for mocks), lifecycle management, cross-cutting concerns via proxies.
  • BeanFactory — minimal, lazy-init, rarely used directly. Base interface.
  • ApplicationContext — extends BeanFactory; adds event publishing, i18n, AOP integration, eager singleton init. Always use this in practice.
  • Common implementations: AnnotationConfigApplicationContext (standalone), WebApplicationContext (servlet container).
"ApplicationContext is BeanFactory plus AOP, events, and eager initialization of singletons on startup — BeanFactory is just the registry."
Dependency Injection types — why constructor wins
Three injection styles — Spring supports all, but they are not equal.
TypeHowVerdict
ConstructorArgs passed at instantiationPreferred — immutable, mandatory deps explicit, fails fast, testable without Spring
Setter@Autowired on setterOK for optional deps; allows partial init
Field@Autowired on private fieldDiscouraged — hides deps, untestable without reflection, circular deps silently accepted
// Constructor injection — preferred
@Service
public class OrderService {
    private final PaymentService payment;

    public OrderService(PaymentService payment) {
        this.payment = payment;
    }
}
Since Spring 4.3, single-constructor class needs no @Autowired — Spring auto-detects it.
Field injection hides the dependency graph. In unit tests you need ReflectionTestUtils or Mockito's @InjectMocks to wire — constructor injection lets you just call new.
Bean lifecycle — full sequence
Order the container follows for every singleton bean:
  1. Instantiate (call constructor)
  2. Populate properties (inject dependencies)
  3. BeanNameAware.setBeanName(), BeanFactoryAware.setBeanFactory(), ApplicationContextAware.setApplicationContext()
  4. BeanPostProcessor.postProcessBeforeInitialization() — runs for ALL beans (e.g., @Autowired processing happens here)
  5. @PostConstruct method (JSR-250)
  6. InitializingBean.afterPropertiesSet()
  7. init-method from XML / @Bean(initMethod=...)
  8. BeanPostProcessor.postProcessAfterInitialization() — where AOP proxies are created
  9. Bean in use
  10. @PreDestroy → DisposableBean.destroy() → destroyMethod
"AOP proxies are wired in postProcessAfterInitialization — that's why a bean's dependencies get the proxy, but this inside the bean does not."
Use @PostConstruct for init logic — it's standard JSR-250, not Spring-specific, so it works in Jakarta EE too.

Singleton + mutable state = race condition. Never store request-specific data in a singleton field. Common bug: injecting a prototype bean into a singleton — you always get the same prototype instance. Fix: inject ApplicationContext and call getBean(), or use @Lookup, or scoped proxy: @Scope(value="prototype", proxyMode=ScopedProxyMode.TARGET_CLASS).
Stereotype annotations — @Component and friends
All four are aliases of @Component — Spring treats them identically for scanning purposes. The distinction is semantic and functional:
AnnotationLayerExtra behavior
@ComponentGenericNone
@ServiceBusiness logicNone (just naming convention)
@RepositoryData accessEnables PersistenceExceptionTranslation — wraps native DB exceptions into Spring's DataAccessException hierarchy
@ControllerMVCMarks class for handler mapping detection
@RestControllerREST@Controller + @ResponseBody
Putting data access code in a @Service loses exception translation — JDBC SQLException won't be wrapped. Use @Repository on DAO classes.
@Autowired, @Qualifier, @Primary — disambiguation
When multiple beans of the same type exist, Spring needs guidance:
  • @Primary — mark one bean as the default candidate. Applied at the bean definition.
  • @Qualifier("beanName") — explicit selection at the injection point. Overrides @Primary.
  • Parameter name matching — without @Qualifier, Spring falls back to matching the parameter name against the bean name.
@Autowired
@Qualifier("asyncMailSender")
private MailSender sender;
Prefer @Primary for "default impl" scenarios (e.g., real vs stub), @Qualifier when you need both simultaneously in different places.
@Autowired(required=false) silently skips injection if no bean found — easy source of NPEs in production. Prefer Optional<T> injection to make optionality explicit.
@Configuration + @Bean — manual wiring
@Configuration marks a class as a bean definition source. Methods annotated with @Bean produce beans managed by the container. 
@Configuration
public class AppConfig {
    @Bean
    public DataSource dataSource() {
        return new HikariDataSource(hikariConfig());
    }

    @Bean
    public HikariConfig hikariConfig() {
        // ...
    }
}
  • Calling hikariConfig() from within @Configuration does NOT call the method again — Spring proxies the class (CGLIB) and returns the existing bean. This is full configuration mode.
  • @Component with @Bean methods = lite mode — no proxy, inter-method calls create new instances. Subtle and dangerous.
Using @Configuration(proxyBeanMethods=false) (lite/performance mode) means cross-method @Bean calls create new instances — only safe if beans don't depend on each other.
"I use @Bean methods in @Configuration when I need third-party library objects — stuff I can't annotate with @Component directly."
Component scanning
Spring scans the classpath for @Component-annotated classes (and its stereotypes) and registers them as beans automatically.
  • @SpringBootApplication = @Configuration + @EnableAutoConfiguration + @ComponentScan — scans the package of the main class and all sub-packages.
  • Custom scan: @ComponentScan(basePackages = "com.example.services")
  • Exclusions: @ComponentScan(excludeFilters = @Filter(type=ASSIGNABLE_TYPE, classes=SomeClass.class))
Placing the main class in the default package (no package) causes Spring to scan the entire classpath — massive startup time and potential conflicts with library internals.
Keep the main class at the root of your package hierarchy so @ComponentScan covers exactly what you own and nothing more.
Circular dependencies — detection and fixes
Circular dependency: A depends on B, B depends on A.
  • Constructor injection: fails fast at startup with BeanCurrentlyInCreationException. This is good — it surfaces design flaws immediately.
  • Field/setter injection: silently resolved by Spring using a three-phase creation cache (early exposure of partially constructed beans). Can mask architectural problems.
Fixes (in order of preference):
  1. Refactor — extract a third bean that both depend on. Root cause is usually an SRP violation.
  2. @Lazy on one injection point — defers creation, breaks the cycle without restructuring.
  3. Setter injection on one side — allows partial construction.
Spring Boot 2.6+ disables circular dependency resolution by default (spring.main.allow-circular-references=false). If your app breaks on upgrade, that's the first thing to check — but fix the root cause, don't just re-enable it.
"Constructor injection making circular deps fail fast is a feature, not a bug — it's the framework enforcing good design."
@Lazy initialization
By default, all singleton beans are initialized eagerly at startup. @Lazy defers initialization until first use.
  • Applied per-bean: @Lazy on @Component or @Bean.
  • Applied globally: spring.main.lazy-initialization=true in application.properties.
  • Applied at injection point: @Autowired @Lazy private HeavyService svc — injects a proxy, real bean created on first method call.
Global lazy init reduces startup time but moves startup failures to runtime — a missing bean won't be caught until the first request hits that code path. Bad for production reliability, useful for development speed and tests.
One legitimate production use: breaking circular dependencies with @Lazy on one injection point, as a targeted fix while refactoring continues.
Proxying — JDK dynamic proxy vs CGLIB, and self-invocation
Spring AOP wraps beans in proxies to apply cross-cutting concerns (@Transactional, @Cacheable, etc.). Two proxy mechanisms:
JDK Dynamic ProxyCGLIB
RequiresInterfaceNo interface needed
HowImplements the interface at runtimeSubclasses the target class at runtime
LimitationOnly interface methods are interceptedCannot proxy final class or final method
DefaultSpring AOP when interface presentSpring Boot default (proxyTargetClass=true)
Self-invocation problem: when a bean calls its own method internally (this.someMethod()), it bypasses the proxy — AOP advice does NOT run. 
@Service
public class InvoiceService {
    @Transactional
    public void processAll() {
        // calls self — @Transactional on save() is IGNORED
        this.save();
    }

    @Transactional(propagation = REQUIRES_NEW)
    public void save() { ... }
}
This is the most common @Transactional bug in interviews. Fix: extract save() to a separate Spring bean, or inject self (@Autowired private InvoiceService self), or use AspectJ weaving instead of proxy-based AOP.
"Spring AOP is proxy-based — the proxy intercepts external calls. Internal calls on this never touch the proxy, so @Transactional, @Cacheable, etc. all silently do nothing."
ApplicationEvents — brief
Spring's internal event bus allows decoupled communication between beans within the same ApplicationContext.
  • Publish: applicationEventPublisher.publishEvent(new OrderCreatedEvent(this, order))
  • Listen: @EventListener on a method, or implement ApplicationListener<T>
  • Async listener: add @Async + @EventListener — runs in a separate thread pool
  • Transactional binding: @TransactionalEventListener(phase = AFTER_COMMIT) — listener fires only after the transaction commits, preventing race conditions where downstream logic runs before DB is consistent
@TransactionalEventListener is the correct pattern for "send email after order is saved" — without it, the email goes out even if the transaction rolls back.
ApplicationEvents are synchronous and in-process by default — not a message broker replacement. For cross-service communication, use Kafka/RabbitMQ.

3. Spring Boot

What Spring Boot adds over plain Spring
Spring Boot is an opinionated wrapper around Spring Framework that eliminates boilerplate configuration via auto-configuration, starter POMs, and an embedded servlet container.
  • Auto-configuration: scans classpath, applies sensible defaults automatically — no XML, no manual bean wiring.
  • Starters: curated dependency sets that guarantee compatible versions.
  • Embedded server: app ships as a fat JAR; no WAR deployment to external Tomcat needed.
  • Production-ready actuators: health, metrics, env endpoints out of the box.
  • Opinionated defaults: Logback for logging, HikariCP for connection pool, Jackson for JSON — all swappable.
"Spring Boot lets me focus on business logic. The framework's opinions handle 90% of infrastructure wiring; I override only what my use case demands."
Interviewers sometimes ask "does Spring Boot replace Spring?" — No. It builds on top. Every Spring Boot app is a Spring app.
@SpringBootApplication — what it really is
@SpringBootApplication is a composed annotation equivalent to three annotations in one:
  • @Configuration — marks the class as a bean definition source.
  • @EnableAutoConfiguration — triggers auto-config mechanism.
  • @ComponentScan — scans the current package and sub-packages for Spring components.
@SpringBootApplication(
    scanBasePackages = "com.example",
    exclude = {DataSourceAutoConfiguration.class}
)
public class App {
    public static void main(String[] args) {
        SpringApplication.run(App.class, args);
    }
}
Default @ComponentScan only scans the package the annotated class lives in and below. If you put the main class in the root package — it scans everything, which can be slow or cause conflicts in large monorepos.
How auto-configuration works
Auto-configuration relies on conditional bean registration. Boot loads candidate configs from:
  • Spring Boot 2.x: META-INF/spring.factories under key EnableAutoConfiguration.
  • Spring Boot 3.x: META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports.
Each auto-config class is annotated with @ConditionalOn* guards:
AnnotationActivates when...
@ConditionalOnClassClass present on classpath
@ConditionalOnMissingBeanNo user-defined bean of that type
@ConditionalOnPropertyProperty key has a specific value
@ConditionalOnWebApplicationRunning as a web app
To see what auto-configs fired and why, start with --debug flag or check the /actuator/conditions endpoint. The "positive matches / negative matches" report is invaluable for debugging.
"Auto-config is conditional — my own beans take precedence. Boot backs off via @ConditionalOnMissingBean."
Overriding auto-configuration
Three ways to override or disable auto-config:
  • Define your own bean: Boot's @ConditionalOnMissingBean makes the auto-config back off. This is the preferred approach.
  • Exclude by class: @SpringBootApplication(exclude = {SecurityAutoConfiguration.class})
  • Exclude via property: spring.autoconfigure.exclude=org.springframework.boot.autoconfigure.security.SecurityAutoConfiguration
// Override HikariCP auto-config by providing own DataSource bean
@Bean
@Primary
public DataSource customDataSource() {
    return DataSourceBuilder.create()
        .url("jdbc:postgresql://...")
        .build();
}
Excluding auto-configs by class name string is error-prone — prefer the class literal form; it fails at compile time if the class doesn't exist.

application.yml vs application.properties
Both are equivalent in Boot's eyes — purely a format preference. YAML is hierarchical and less repetitive; properties is flat key=value. 
# application.yml
spring:
  datasource:
    url: jdbc:postgresql://localhost/mydb
    username: app
  jpa:
    hibernate:
      ddl-auto: validate

# application.properties equivalent
spring.datasource.url=jdbc:postgresql://localhost/mydb
spring.datasource.username=app
spring.jpa.hibernate.ddl-auto=validate
  • YAML supports multi-document files (separated by ---) — useful for embedding multiple profiles in one file.
  • Properties does not support lists as naturally as YAML.
  • If both exist, properties wins over YAML for the same key — they merge, not replace.
YAML is whitespace-sensitive. A tab instead of spaces causes a parse error at startup with a misleading message. Always use spaces.
Profiles — @Profile, activation, profile files
Profiles let you activate different bean sets and config values per environment.
  • @Profile("prod") on a @Bean or @Configuration — only loaded when that profile is active.
  • Activate via: spring.profiles.active=prod in properties, env var SPRING_PROFILES_ACTIVE=prod, or JVM arg -Dspring.profiles.active=prod.
  • Profile-specific files: application-prod.yml, application-dev.yml — auto-loaded and merged over the base application.yml.
  • Spring Boot 2.4+ replaced spring.profiles with spring.config.activate.on-profile for multi-doc YAML.
# application-prod.yml
spring:
  datasource:
    url: jdbc:postgresql://prod-db:5432/app
logging:
  level:
    root: WARN
"In Kubernetes we set SPRING_PROFILES_ACTIVE=prod as an env var in the deployment spec — no config inside the image."
@Profile("!prod") means "active on all profiles EXCEPT prod". Useful for dev-only beans like test data seeders — make sure you don't accidentally exclude them from staging.
Externalized configuration — property source order
Spring Boot resolves properties in a strict precedence order (higher overrides lower):
  1. Command-line arguments (--server.port=9090)
  2. JNDI attributes (java:comp/env)
  3. Java System properties (-Dkey=value)
  4. OS environment variables (SERVER_PORT=9090)
  5. Profile-specific files outside jar (application-prod.yml in file system)
  6. Profile-specific files inside jar
  7. application.yml / application.properties outside jar
  8. application.yml / application.properties inside jar
  9. @PropertySource annotations
  10. Default properties
In GCP/Kubernetes, env vars are the standard injection vector. Spring's relaxed binding maps SPRING_DATASOURCE_URL to spring.datasource.url automatically — underscores map to dots, all uppercase accepted.
Files outside the jar (same directory or config/ subdirectory) override files inside the jar. Misconfigured deployments with stale config files on the filesystem silently win over the packaged config.

@RestController vs @Controller
@RestController = @Controller + @ResponseBody on every method — return value goes straight through HttpMessageConverter, not to a view resolver.
  • Use @Controller when mixing MVC views and REST endpoints in one class — rare in microservices.
  • Use @RestController for every pure API controller.
  • @ResponseBody can be added per-method on a @Controller class when one method needs view resolution and another needs JSON.
Interviewer asks: "What does @RestController actually do?" Answer at the annotation composition level — not just "returns JSON". The converter chain (Jackson by default) is what produces JSON; the annotation just bypasses view resolution.
@RestController
@RequestMapping("/api/orders")

Mapping annotations (@GetMapping, @PostMapping, etc.)
All are composed shortcuts for @RequestMapping(method = RequestMethod.X).

  • @RequestBody — deserializes HTTP body via HttpMessageConverter. Used for POST/PUT/PATCH payloads. Pair with @Valid for bean validation.

ResponseEntity — when to use vs bare return
ResponseEntity<T> gives full control over status code, headers, and body. Use when:
  • Creating a resource — return 201 Created + Location header.
  • No content response — return 204 No Content with empty body.
  • Conditional responses — 304 Not Modified with ETag logic.
  • Setting custom response headers.
Bare return (just the DTO) — acceptable for simple 200 OK GETs. Cleaner, less boilerplate. 
@PostMapping
public ResponseEntity<OrderDto> create(@Valid @RequestBody CreateOrderRequest req) {
    OrderDto dto = orderService.create(req);
    URI location = ServletUriComponentsBuilder.fromCurrentRequest()
        .path("/{id}").buildAndExpand(dto.id()).toUri();
    return ResponseEntity.created(location).body(dto);
}
"I return ResponseEntity when I need to control the status or set headers — like 201 + Location on create. For standard reads I return the DTO directly and let Spring default to 200."

REST design principles
  • Resources, not actions — POST /orders not POST /createOrder.
  • Plural nouns — /orders, /users/{id}/orders.
  • HTTP verb carries the intent — don't put the verb in the URL.
  • Idempotency: GET, PUT, DELETE must be idempotent. POST is not. PATCH — design-dependent.
  • Stateless: each request carries all context. No server-side session.
  • Nested resources sparingly — max 2 levels deep (/users/{id}/orders). Deeper = fragile.
  • Filtering/sorting/pagination via query params, not path segments.
For actions that don't map cleanly to CRUD (e.g. cancel, approve), use a sub-resource noun: POST /orders/{id}/cancellation rather than POST /orders/{id}/cancel.
"Idempotency means calling the same request N times has the same effect as calling it once. GET never changes state. PUT replaces the full resource to the same target state. DELETE on an already-deleted resource should return 404 or 204 — either is defensible, but be consistent."
Validation (@Valid, @Validated, jakarta.validation)
Spring integrates Hibernate Validator (Jakarta Bean Validation 3.x).
  • @Valid on method parameter triggers validation; violations throw MethodArgumentNotValidException → 400.
  • Common constraints: @NotNull, @NotBlank, @Size(min,max), @Min, @Max, @Pattern(regexp), @Email, @Positive.
  • @Validated on the class enables method-level validation and supports constraint groups — different rules for create vs update.
  • Nested objects need @Valid on the field to cascade validation.

Global error handling (@ControllerAdvice + Problem Details)
@ControllerAdvice is an AOP-based mechanism to handle exceptions globally across all controllers.
  • @ExceptionHandler(SomeException.class) — handles specific exception type.
  • Return ResponseEntity<ProblemDetail> for RFC 7807 compliance.
  • Spring Boot 3.x has built-in RFC 7807 support — enable with spring.mvc.problemdetails.enabled=true.
@ControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(OrderNotFoundException.class)
    public ResponseEntity<ProblemDetail> handleNotFound(OrderNotFoundException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Order Not Found");
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(pd);
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ProblemDetail> handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(HttpStatus.BAD_REQUEST);
        pd.setDetail("Validation failed");
        pd.setProperty("errors", ex.getBindingResult().getFieldErrors()
            .stream().map(e -> e.getField() + ": " + e.getDefaultMessage()).toList());
        return ResponseEntity.badRequest().body(pd);
    }
}
RFC 7807 ProblemDetail fields: type (URI), title, status, detail, instance. Clients can parse errors predictably. Standard in Spring Boot 3+.

  • Default: Jackson's MappingJackson2HttpMessageConverter handles JSON. Add jackson-dataformat-xml to support XML automatically.
@PostMapping(consumes = "application/json", produces = "application/json")
public ResponseEntity<OrderDto> create(@RequestBody CreateOrderRequest req) { ... }
Being explicit with produces/consumes documents the contract in the code and prevents accidental content type mismatches.

5. Spring Data / JPA / Hibernate

Layer architecture: JPA vs Hibernate vs Spring Data
JPA is a specification (javax.persistence / jakarta.persistence). Hibernate is the implementation. Spring Data JPA is an abstraction layer on top that generates repository implementations at runtime.
  • Spring Data JPA delegates to a JPA EntityManager backed by Hibernate under the hood.
  • Hibernate manages the persistence context (unit of work, first-level cache, dirty checking).
  • You can drop to native Hibernate API via entityManager.unwrap(Session.class) when needed.
"JPA is the interface contract, Hibernate is the engine, Spring Data is the scaffolding that wires them together and generates boilerplate."
Repository hierarchy
Each interface adds capabilities:

Prefer JpaRepository in most Spring Boot apps. Use CrudRepository if you want to avoid pulling in JPA-specific methods in a module boundary.
getReferenceById returns a proxy without hitting the DB — throws EntityNotFoundException on field access if entity doesn't exist. findById hits DB immediately and returns Optional.
Derived query methods and @Query
Spring Data parses method names and generates JPQL automatically:

Keywords: By, And, Or, Between, Like, In, OrderBy, Asc/Desc, IsNull, Distinct.

@Query — explicit JPQL (entity-based) or native SQL: 
// JPQL — uses entity/field names, not table/column names
@Query("SELECT u FROM User u WHERE u.email = :email")

// Native SQL — uses real table/column names

@Modifying required for UPDATE/DELETE queries; pair with clearAutomatically = true to evict stale entities from context:

@Query("UPDATE User u SET u.status = :s WHERE u.id = :id")

JPQL uses entity class names and field names. Forgetting this and writing SQL column names is a common mistake — silently fails or throws.

Fetch types and N+1 problem
Default fetch types — interviewers probe this constantly:
AnnotationDefault Fetch
@ManyToOneEAGER
@OneToOneEAGER
@OneToManyLAZY

N+1 problem: loading a list of N orders triggers N additional queries to load each order's customer (EAGER) or accessing a lazy collection in a loop triggers N queries.

How to detect: enable SQL logging (spring.jpa.show-sql=true + format_sql), or Hibernate statistics (hibernate.generate_statistics=true), or p6spy.

How to fix:
  • JOIN FETCH in JPQL: SELECT o FROM Order o JOIN FETCH o.items WHERE o.id = :id
  • @EntityGraph: declarative, avoids modifying query method:
@EntityGraph(attributePaths = {"items", "customer"})
List<Order> findByStatus(Status status);
  • @BatchSize on collection: fetches in chunks (SELECT ... WHERE id IN (...))
  • DTO projection: only select what you need — no entity loading at all
JOIN FETCH with pagination (Pageable) causes Hibernate to load all rows into memory and paginate in-app — it logs a warning. Use @BatchSize or a two-query approach instead.
@Transactional — propagation, rollback, self-invocation
Propagation types:
PropagationBehavior

REQUIRES_NEWSuspend current tx, start new one; commits independently
NESTEDSavepoint inside current tx; rollback to savepoint on failure

Rollback rules: by default only RuntimeException and Error trigger rollback. Checked exceptions do NOT. 
@Transactional(rollbackFor = Exception.class)  // include checked exceptions
public void process() throws IOException { ... }
Self-invocation problem: @Transactional works via Spring proxy. Calling a @Transactional method from within the same bean bypasses the proxy — no transaction started. 
// BROKEN — this.doWork() bypasses proxy
public void outer() { this.doWork(); }

@Transactional
public void doWork() { ... }
Fix: inject the bean into itself (@Autowired ApplicationContext), use AspectJ weaving, or restructure into separate beans.
"Self-invocation is the most common @Transactional gotcha in Spring — the proxy never intercepts it."
Persistence context and first-level cache
The persistence context is a unit of work scoped to a transaction by default (PersistenceContextType.TRANSACTION). It acts as the first-level cache — load the same entity twice by ID within one transaction: only one DB query.

Entity states:
  • Transient — new object, not tracked
  • Managed — tracked by persistence context; changes auto-flushed at commit
  • Detached — was managed, transaction ended; changes not tracked
  • Removed — scheduled for DELETE
Dirty checking: at flush time Hibernate snapshots vs. current state and issues UPDATE only for changed fields.

Flush modes: AUTO (default) — flush before query execution and at commit. COMMIT — only at commit. MANUAL — never automatic.
LazyInitializationException — accessing a lazy collection/association after the transaction (and persistence context) has closed. Fix: ensure access within transaction, use JOIN FETCH, @Transactional on service layer, or DTO projection.
Optimistic vs pessimistic locking
Optimistic locking — assume no conflict; detect at commit. Uses a @Version field (integer or timestamp): 
@Entity
public class Product {
    @Version
    private Long version;
    // ...
}
Hibernate appends WHERE id = ? AND version = ? to UPDATE. If 0 rows updated → throws OptimisticLockException. No DB lock held.

Pessimistic locking — acquire DB lock upfront: 
// Issues SELECT ... FOR UPDATE
productRepo.findById(id, LockModeType.PESSIMISTIC_WRITE);

// Or via @Lock on repository method
@Lock(LockModeType.PESSIMISTIC_READ)
Optional<Product> findById(Long id);
OptimisticPessimistic
DB lockNoYes (row-level)
Best forLow contention, read-heavyHigh contention, short critical section
Failure pointOptimisticLockException at commitDeadlock risk, timeout
Prefer optimistic locking for most web APIs. Pessimistic locking is warranted for inventory/booking systems where conflicts are frequent and retry cost is high.
Relationships, cascading, and orphanRemoval
Cascade types: PERSIST, MERGE, REMOVE, REFRESH, DETACH, ALL. 
@OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
private List<OrderItem> items = new ArrayList<>();
  • cascade = REMOVE — deleting parent deletes children via Hibernate (individual DELETEs per child)
  • orphanRemoval = true — removing item from items collection schedules it for DELETE; implies REMOVE cascade but also handles collection removal
  • deleteAllInBatch() on repo — single SQL DELETE, bypasses cascade entirely — can break FK constraints if children exist
Bidirectional consistency: both sides of a bidirectional relationship must be kept in sync manually in Java; Hibernate only reads the owning side (the one without mappedBy). 
order.getItems().add(item);
item.setOrder(order); // Must set both sides
@ManyToMany with cascade = REMOVE or ALL — removing one side cascades deletes to the other entity, not just the join table. Almost always wrong. Only cascade PERSIST and MERGE on many-to-many.
DTO projections (interface and class)
Loading full entities for read-only operations wastes memory and triggers lazy loading pitfalls. Spring Data supports projections:

Interface projection — Spring generates a proxy: 
public interface UserSummary {
    String getEmail();
    String getFullName();
}

Class (DTO) projection — use constructor expression in JPQL: 
@Query("SELECT new com.example.dto.UserDto(u.id, u.email) FROM User u WHERE u.status = :s")

Or with Spring Data @Value on interface fields (SpEL supported).

Native query projections — interface projection works with nativeQuery = true; column names must match getter names (case-insensitive via alias).
For complex reporting queries returning many rows, class projections with native SQL are the most performant option — no entity lifecycle overhead, no dirty checking snapshot.
Interface projection on a nested association (e.g., getAddress().getCity()) still triggers a join or separate query — it's not automatically flat. Explicit @Query with JOIN is safer.

Common interview traps quick-fire
  • Checked exceptions don't rollback — must use rollbackFor = Exception.class or rollbackFor = YourCheckedException.class.
  • @Transactional on private methods — Spring proxy can't intercept private methods; annotation silently ignored (no exception thrown).
  • save() on a managed entity — inside a transaction, a managed entity is already dirty-checked; calling save() is a no-op (merge of managed entity). Flushed at commit anyway.
  • findAll() on large table — loads entire table into memory. Always use findAll(Pageable) or streams with @QueryHints(ScrollableResultsConsumer).
  • equals()/hashCode() on entities — must be based on business key or ID (but ID is null before persist). Use natural key if possible. Hibernate-generated proxies break identity-based equals.
  • open-in-view anti-pattern — Spring Boot enables spring.jpa.open-in-view=true by default; keeps persistence context open through HTTP request including view rendering. Hides N+1 but ties DB connections to web layer. Disable in production.
"I'd disable open-in-view in any production app — it trades convenience for hidden query leakage and longer connection hold times."

SQL Execution Order
The logical execution order of a SELECT — not the write order — determines what you can reference where.
StepClauseNote
1FROM / JOINbuild working set

3GROUP BYcollapse to groups
4HAVINGfilter groups — can use aggregates

You cannot use a SELECT alias in WHERE or HAVING — it doesn't exist yet at those steps. Use a subquery or CTE instead.
"WHERE filters individual rows before grouping; HAVING filters aggregated groups after — that's why aggregate functions belong in HAVING, not WHERE."

FULL OUTER JOIN ≠ UNION. UNION stacks rows vertically; FULL OUTER JOIN merges horizontally with NULLs.
Subqueries vs CTEs (WITH)
A subquery is a nested SELECT inside another statement. A CTE (Common Table Expression) is a named temporary result set defined before the main query with WITH.
  • CTEs improve readability — break complex logic into named steps.
  • In PostgreSQL, a non-recursive CTE is inlined by the planner (since PG 12) — same performance as a subquery unless you add MATERIALIZED.
  • Recursive CTE — only way to walk hierarchies in SQL:
WITH RECURSIVE subordinates AS (
  SELECT id, name, manager_id FROM employee WHERE id = 1
  UNION ALL
  SELECT e.id, e.name, e.manager_id
  FROM employee e
  JOIN subordinates s ON e.manager_id = s.id
)
SELECT * FROM subordinates;
Use WITH ... AS MATERIALIZED to force Postgres to evaluate a CTE once and cache — useful when the same CTE is referenced multiple times.
Correlated subqueries run once per row of the outer query — O(n) hidden cost. Rewrite as JOIN or lateral join where possible.
Window Functions
Window functions compute a value across a set of rows related to the current row without collapsing them (unlike GROUP BY aggregates). 
SELECT
  name, dept, salary,
  ROW_NUMBER() OVER (PARTITION BY dept ORDER BY salary DESC) AS rn,
  RANK()       OVER (PARTITION BY dept ORDER BY salary DESC) AS rnk,
  DENSE_RANK() OVER (PARTITION BY dept ORDER BY salary DESC) AS drnk,
  LAG(salary)  OVER (PARTITION BY dept ORDER BY salary DESC) AS prev_sal,
  LEAD(salary) OVER (PARTITION BY dept ORDER BY salary DESC) AS next_sal
FROM employee;
FunctionTies behavior
ROW_NUMBERalways unique — arbitrary tiebreak
RANKgaps after ties (1,1,3)
DENSE_RANKno gaps (1,1,2)
  • PARTITION BY — resets the window per group (like GROUP BY but keeps all rows).
  • LAG / LEAD — access previous/next row value; great for period-over-period delta.
  • SUM() OVER (ORDER BY date ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) — running total.
"For 'find the top earner per department' I'd use DENSE_RANK in a CTE then filter WHERE drnk = 1 — cleaner than a correlated subquery."
Indexes — Types and Strategies
An index is a separate data structure keeping a sorted reference to rows, trading write overhead for read speed.

Multi-column index — leftmost prefix rule: index on (a, b, c) supports queries filtering on a, a,b, or a,b,c — but NOT b alone.

Covering index (INCLUDE): columns added to index leaf pages — query satisfied entirely from index (index-only scan): 
CREATE INDEX idx_orders_covering
ON orders (customer_id)
INCLUDE (total, created_at);
Partial index: index only rows matching a predicate — smaller, faster:

Every index slows INSERT / UPDATE / DELETE. On write-heavy tables, benchmark before adding.
Unused indexes: pg_stat_user_indexes.idx_scan = 0 after steady traffic — drop them.
EXPLAIN vs EXPLAIN ANALYZE
EXPLAIN shows the planner's estimated execution plan — no query is actually run. EXPLAIN ANALYZE executes the query and shows actual runtime vs estimates. 
EXPLAIN ANALYZE SELECT * FROM orders WHERE customer_id = 42;
Key nodes to recognize:

  • Cost unit is arbitrary (disk page fetches). Compare relative costs, not absolute numbers.
EXPLAIN ANALYZE on a write query (INSERT/UPDATE/DELETE) actually executes it. Wrap in a transaction and rollback, or use EXPLAIN (ANALYZE, BUFFERS) in a test environment.
"I see a Seq Scan on a 10M-row table — first thing I check is whether the column has an index and whether the planner is using it; if not, I look at selectivity and statistics."
Transactions, ACID, and Isolation Levels
ACID: Atomicity (all-or-nothing), Consistency (constraints held), Isolation (concurrent txns don't see each other's dirt), Durability (committed = persisted).
LevelDirty ReadNon-repeatable ReadPhantom ReadPG behavior
READ UNCOMMITTEDallowed (spec)yesyesPostgres treats as READ COMMITTED
READ COMMITTEDnoyesyesdefault in PG
REPEATABLE READnonono (PG blocks phantoms too)snapshot at txn start

  • Dirty read — seeing uncommitted data from another txn.
  • Non-repeatable read — same row returns different value within same txn.
  • Phantom read — same range query returns different rows within same txn.
PostgreSQL has no READ UNCOMMITTED in practice — it silently upgrades to READ COMMITTED. Mentioning this in an interview signals you actually know Postgres, not just theory.
"READ COMMITTED is the default — fine for most OLTP. For financial operations where I need repeatable reads, I'd use REPEATABLE READ or SERIALIZABLE, accepting potential serialization errors that I need to retry."
MVCC, VACUUM, and Bloat
MVCC (Multi-Version Concurrency Control) — each transaction sees a snapshot of the database at a point in time. Postgres never overwrites rows in-place; UPDATE writes a new row version (tuple) and marks the old one dead. No read locks needed.
  • Dead tuples accumulate = table bloat — wasted disk, slower scans.
  • VACUUM marks dead tuples as reusable space (doesn't shrink the file).
  • VACUUM FULL rewrites the table — reclaims disk, but takes an exclusive lock. Avoid in production during business hours.
  • autovacuum runs automatically; tune autovacuum_vacuum_scale_factor for large tables (default 20% is too high — set to 1-2% for multi-million-row tables).
Monitor bloat: pg_stat_user_tables.n_dead_tup. If dead tuples >> live tuples after autovacuum, tune thresholds or run manual VACUUM.
Long-running transactions prevent VACUUM from cleaning dead tuples older than the transaction's snapshot. A forgotten open transaction is a classic source of runaway bloat.
Connection Pooling — HikariCP and PgBouncer
PostgreSQL spawns a process per connection — expensive. Connection pooling reuses connections.
  • HikariCP — Spring Boot default (spring.datasource.hikari.*). Pool lives inside the JVM. Handles one app instance.
  • PgBouncer — external proxy, sits between app and Postgres. Use when multiple app instances or microservices would otherwise open hundreds of connections. Modes: session (safe, most compatible), transaction (most efficient), statement.
Pool sizing rule of thumb: pool_size = num_cores * 2 + effective_spindle_count (Hikari documentation). For a 4-core machine: ~9. More connections ≠ more throughput — Postgres will just context-switch. 
# application.yml
spring.datasource.hikari:
  maximum-pool-size: 10
  minimum-idle: 5
  connection-timeout: 30000
  idle-timeout: 600000
Setting pool size to 100 "to be safe" causes contention. Postgres has a max_connections default of 100 — shared across all apps. Always check the server limit before sizing the app pool.
"In a Spring Boot microservice I use HikariCP sized to ~10 connections. If we have 20 service instances hitting the same Postgres, I'd put PgBouncer in transaction mode in front to keep total server connections manageable."

UPSERT, Sequences, and Constraints
UPSERT — INSERT ... ON CONFLICT: 
INSERT INTO users (id, email, name)
VALUES (1, 'a@b.com', 'Alice')
ON CONFLICT (id) DO UPDATE
  SET email = EXCLUDED.email, name = EXCLUDED.name;
-- EXCLUDED refers to the row that would have been inserted
Use ON CONFLICT DO NOTHING to silently skip duplicates.

Sequences / SERIAL / IDENTITY:
  • SERIAL — legacy shorthand; creates a sequence under the hood.
  • GENERATED ALWAYS AS IDENTITY — SQL standard, preferred in PG 10+.
  • Sequences are not rolled back on transaction failure — gaps are normal and expected.

Performance anti-patterns to mention:
  • SELECT * — pulls unnecessary columns, prevents index-only scans.
  • IN (SELECT ...) with large subquery — rewrite as JOIN or EXISTS.
  • Functions on indexed columns in WHERE: WHERE LOWER(email) = 'x' disables the index — create a functional index or store data normalized.
  • N+1 queries from ORM — detect with slow query log or pg_stat_statements.
  • Batch inserts over row-by-row inserts — use COPY or multi-row VALUES for bulk loads.
"For bulk inserts in Spring I'd use JdbcTemplate batchUpdate or Spring Data's saveAll — but for very large volumes I'd consider COPY via the PgCopy API or a staging table."

7. Microservices Patterns

Monolith vs Microservices — when NOT to split
Microservices trade deployment independence for distributed systems complexity. The split only pays off when teams, traffic, and platform are ready.

Don't split when:
  • Team < 5 engineers — communication overhead eats the benefit
  • No CI/CD pipeline, no container platform — you'll deploy a monolith more reliably
  • Domain not understood yet — wrong cuts create chatty services worse than a monolith
  • No observability infrastructure — debugging across 10 services without tracing is hell
Do split when: independent deploy cycles per team, genuinely different scaling needs per component, regulatory isolation.
Trap: "microservices = modern". Interviewers probe whether you've felt the operational pain. Say: distributed monolith is the worst of both worlds — split services that still share a DB or deploy together.
"Microservices solve an organizational problem as much as a technical one. Conway's Law: your architecture mirrors your team structure."
Bounded Contexts and DDD
A Bounded Context is the explicit boundary within which a domain model is consistent and unambiguous. Each microservice should own exactly one bounded context.

Key DDD building blocks:
  • Aggregate — cluster of domain objects treated as one unit; only the aggregate root is addressed from outside
  • Entity — identity-based object; Value Object — equality by value, immutable
  • Domain Event — something that happened in the domain (past tense: OrderPlaced)
  • Anti-corruption Layer (ACL) — translator at context boundaries; prevents alien model leaking in
Map bounded contexts before drawing service boxes. Wrong context boundaries = chatty RPCs and coupled deploys — structural debt that compounds.
Trap: "User" means something different in Billing vs Shipping. Same word, different model — that's where context boundaries belong.
Database per Service — the rule and the pain
Each service owns its data store exclusively. No other service queries it directly — only via the service's API. This preserves independent deployability and encapsulation.

Pain it brings:
  • No cross-service JOINs — must denormalize or compose in application layer
  • Eventual consistency instead of ACID across services
  • Data duplication — e.g., Order service caches product name at order time
  • Cross-service queries → API calls → latency, partial failures
  • Reporting layer needs a read model (event-sourced projection, warehouse)
Shared database anti-pattern: services coupled at schema level — one team's migration breaks another's service. Defeats the whole point.
For reporting across services, use an event-driven read model: services emit events, a dedicated reporting service materializes an aggregated view.
Service Discovery — client-side vs server-side
Services need to find each other without hardcoded IPs. Two patterns:

ApproachHow it worksExample
Client-sideClient queries registry, gets list, picks instance (load balances itself)Netflix Eureka + Spring Cloud LoadBalancer
Server-sideClient hits a fixed endpoint; router/LB resolves internallyk8s Service + kube-proxy; AWS ALB

On Kubernetes (GCP GKE): server-side via ClusterIP Services — DNS resolves my-service.namespace.svc.cluster.local. Eureka unnecessary in k8s.
On GCP/k8s, lean on k8s service discovery + Cloud Load Balancing. Eureka is a Spring-Cloud-on-VMs pattern — know it for legacy interview context, not as a greenfield choice.
API Gateway — Spring Cloud Gateway
The gateway is the single entry point from outside to internal services. It handles cross-cutting edge concerns so individual services don't have to.

Responsibilities:
  • Routing — path/header predicates to downstream services
  • Authentication/Authorization — validate JWT, forward identity header
  • Rate limiting — RequestRateLimiter filter (Redis-backed)
  • SSL termination, CORS
  • Request aggregation — BFF pattern: one gateway call fans out to N services
  • Circuit breaking — via Resilience4j filter
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
Trap: putting business logic in the gateway. It's infrastructure — routing, auth, rate limits only. Business rules belong in services.
Service Mesh — Istio / Envoy (sidecar approach)
A service mesh moves cross-cutting network concerns (mTLS, retries, tracing, traffic shaping) out of application code into a sidecar proxy (Envoy) injected per pod.

What Istio gives you:
  • mTLS — automatic mutual TLS between services, zero-trust network
  • Retries, timeouts, circuit breaking — configured in VirtualService / DestinationRule, not in app code
  • Traffic splitting — canary, A/B at percentage level
  • Distributed tracing — Envoy injects trace headers automatically
  • Telemetry — metrics scraped from Envoy, not app
Library approach (Resilience4j, Spring Cloud): resilience logic in code, language-specific, simpler ops but per-service config.
Mesh shines at scale with polyglot services. For a Java-only shop on GKE, it adds operational overhead — weigh carefully. GCP has Traffic Director / Anthos Service Mesh.

Rule of thumb: use sync when the caller needs the response to proceed (query, command needing ack). Use async when: event notification, fire-and-forget, fan-out to multiple consumers, long-running processing.
Trap: long synchronous chains A→B→C→D. One slow service stalls the whole chain. Convert to async or at minimum add timeouts + circuit breakers at every hop.
"In GCP context, Pub/Sub for async eventing; Kafka when you need replay, consumer groups, or ordering guarantees per partition."
Circuit Breaker — Resilience4j states
A circuit breaker wraps a call to a remote service. When failures exceed a threshold, it opens and fast-fails subsequent calls — preventing cascade failures.

Three states:
  • CLOSED — calls pass through; failures counted in sliding window
  • OPEN — calls fail immediately (fallback executed); waits waitDurationInOpenState
  • HALF_OPEN — limited probe calls allowed; if they succeed → CLOSED; if fail → OPEN again
@CircuitBreaker(name = "inventoryService", fallbackMethod = "inventoryFallback")
public InventoryResponse checkStock(String productId) {
    return inventoryClient.check(productId);
}

private InventoryResponse inventoryFallback(String productId, Exception ex) {
    return InventoryResponse.unavailable(); // degrade gracefully
}
Metrics to tune: slidingWindowSize, failureRateThreshold, slowCallRateThreshold, waitDurationInOpenState.
Hystrix is deprecated — do not mention it as your current choice. Resilience4j is the answer (also Sentinel for Alibaba stack).
Retry, Backoff, Jitter, Bulkhead, Timeout
Retry: transient failures (network blip, 503) — retry with limit. Never retry on 4xx (client error) or non-idempotent ops without idempotency keys.

Exponential backoff: wait = base * 2^attempt. Without jitter, all retrying clients fire simultaneously — thundering herd. Add jitter: wait = random(0, base * 2^attempt).

Bulkhead: isolate resource pools per downstream. If Service A has a thread pool of 10 for calls to Service B, a B outage exhausts only those 10 threads — the rest of A continues serving other requests. 
Bulkhead bulkhead = Bulkhead.of("inventory",
    BulkheadConfig.custom().maxConcurrentCalls(10).build());
Timeout: always set explicit timeouts on every outbound HTTP call. Default = infinite = thread held forever on dead downstream. 
// WebClient
webClient.get().uri(url)
    .retrieve()
    .bodyToMono(Response.class)
    .timeout(Duration.ofSeconds(3));
Trap: retry without idempotency key on POST. Can create duplicate orders. Always pair retries with idempotency keys on mutations.
Distributed Transactions — SAGA, Outbox (no 2PC)
2PC across services = don't. It requires a coordinator, blocks resources across services, and a coordinator crash leaves participants in uncertainty. Kills availability.

SAGA pattern — sequence of local transactions, each publishing an event/message. Compensating transactions for rollback.
ChoreographyOrchestration
CoordinatorNone — services react to eventsCentral orchestrator service
CouplingLoose (event-based)Orchestrator knows all steps
DebuggingHard — flow is implicitEasier — flow in one place
Best forSimple flowsComplex, long-running sagas

Outbox pattern — reliably publish events: write domain event to outbox table in same DB transaction as domain change. A separate poller/CDC (Debezium) reads outbox and publishes to Kafka. Guarantees at-least-once delivery without distributed transaction. 
// In same transaction:
orderRepo.save(order);
outboxRepo.save(new OutboxEvent("OrderPlaced", order.getId(), payload));
// Debezium picks up outbox row via CDC and publishes to Kafka
"Eventual consistency is not a bug. Design compensating flows and make idempotent consumers — that's production-grade distributed thinking."
Distributed Tracing and Observability Stack
Distributed tracing — correlate a request across N services via a shared trace-id. Each service creates a span. Spans form a tree (the trace).

W3C traceparent header — standard propagation format: 00-traceId-spanId-flags. OpenTelemetry auto-instruments Spring Boot and propagates it.

Stack:
  • OpenTelemetry — vendor-neutral instrumentation SDK (replaces Sleuth in Spring Boot 3)
  • Zipkin / Jaeger — trace backends for storage and UI
  • Micrometer — metrics facade; expose via Prometheus endpoint (/actuator/prometheus)
  • Prometheus + Grafana — scrape, alert, visualize
  • ELK / Loki — centralized logs; structured JSON logs with traceId field for correlation
Always log traceId in every log line. In Spring Boot 3 + Micrometer Tracing: MDC is populated automatically — just use SLF4J and it appears in JSON output.
On GCP: Cloud Trace, Cloud Logging, Cloud Monitoring are the managed equivalents — know both OSS and GCP-native.
Health Checks — Liveness vs Readiness (k8s)
Liveness probe: "Is the process alive?" — if it fails, k8s kills and restarts the pod. Use for deadlocks, corrupted state. Should NOT check external dependencies (DB down ≠ restart the pod).

Readiness probe: "Is it ready to serve traffic?" — if it fails, k8s removes pod from load balancer endpoints. Use for startup warmup, downstream dependency unavailable.

Spring Boot Actuator maps both automatically: 
management:
  endpoint:
    health:
      probes:
        enabled: true
  health:
    livenessState:
      enabled: true
    readinessState:
      enabled: true
# Endpoints: /actuator/health/liveness  /actuator/health/readiness
Trap: liveness probe hitting the DB. DB outage → liveness fails → pod restarted → still can't reach DB → restart loop. Liveness = app process health only. Readiness = external deps.
Config Management and API Versioning
Config sources (12-factor: config in environment):
  • Spring Cloud Config Server — centralized Git-backed config, supports profiles and label
  • k8s ConfigMap / Secret — mounted as env vars or files
  • GCP Secret Manager — managed secrets, IAM-controlled, audit log
Never bake secrets into images. Rotate via Secret Manager versions; app reads at startup or via dynamic refresh (@RefreshScope).

API versioning — additive change strategy:
  • Prefer URI versioning: /api/v1/orders vs /api/v2/orders
  • Make changes additive: add fields, don't remove/rename existing ones
  • Deprecate old version with Deprecation / Sunset HTTP headers
  • Run v1 and v2 in parallel until consumers migrate
  • Consumer-driven contract testing (Pact) — producers can't break consumers silently
Trap: renaming a JSON field in a shared event schema. Downstream consumers break silently at runtime — no compile-time safety. Schema registry + Avro/Protobuf enforces compatibility.
When Microservices Hurt — the honest answer
Interviewers with real experience ask this. Have a straight answer.

Real operational pain:
  • Shared library hell — common commons-dto jar couples all services; every release needs coordinated deploy. Library = hidden monolith.
  • Schema evolution — changing event schema breaks consumers. Requires schema registry, versioning, backward/forward compatibility discipline.
  • Debugging across hops — a 500ms latency spike somewhere in A→B→C→D requires full tracing stack. Without it, you're guessing.
  • Local development complexity — running 10 services locally. Requires docker-compose, Testcontainers, or a dev cluster.
  • Distributed transactions — SAGA compensations are hard to get right. Partial failures leave inconsistent state that needs reconciliation jobs.
  • Operational overhead — each service needs its own: CI/CD, health checks, alerting, log aggregation, secret management.
"The question isn't monolith vs micro — it's: does our team's maturity and platform justify the operational tax? For a small team without a platform, the answer is usually no."

JUnit 5 Lifecycle & Key Annotations
JUnit 5 (Jupiter) — annotations live in org.junit.jupiter.api.

  • @BeforeEach / @AfterEach — run before/after EACH test method; new instance per test by default.
  • @BeforeAll / @AfterAll — run once per class; method must be static (unless @TestInstance(PER_CLASS)).

  • @Nested — inner class grouping related scenarios; has its own lifecycle.

Assertions: JUnit 5 vs AssertJ
JUnit 5 built-ins: assertEquals, assertNotNull, assertThrows, assertAll. AssertJ is preferred in modern Spring projects — fluent, readable, better failure messages. 
// JUnit 5
assertThrows(IllegalArgumentException.class, () -> service.process(null));
assertAll(
    () -> assertEquals("Alice", user.getName()),
    () -> assertEquals(30, user.getAge())
);

// AssertJ
assertThat(result).isNotNull()
                  .extracting(User::getName)
                  .isEqualTo("Alice");
assertThatThrownBy(() -> service.process(null))
    .isInstanceOf(IllegalArgumentException.class)
    .hasMessageContaining("null input");
  • assertAll — runs all assertions even if one fails (grouped failures). Use when validating multiple fields.
  • AssertJ: contains, containsExactly, hasSize, satisfies, usingRecursiveComparison() for deep object equality without equals().
Use usingRecursiveComparison() instead of implementing equals in DTOs just for tests.
Mockito: Core API
Mockito creates test doubles at runtime via bytecode proxies.
  • @ExtendWith(MockitoExtension.class) — activates annotation-based injection in JUnit 5 (no MockitoAnnotations.openMocks(this) needed).
  • @Mock — creates a mock; all methods return defaults (null / 0 / empty list).
  • @InjectMocks — creates the class under test and injects mocks into it (constructor → setter → field injection, in that order).
  • when(mock.method(arg)).thenReturn(value) — stub a call.
  • thenThrow(new SomeException()) — stub exception.
  • verify(mock).method(arg) — assert the method was called.
  • verify(mock, times(2)), never(), atLeastOnce().
@Mock UserRepository repo;
@InjectMocks UserService service;

@Test
void findUser_returnsMappedDto() {
    when(repo.findById(1L)).thenReturn(Optional.of(new User(1L, "Alice")));
    UserDto dto = service.findUser(1L);
    assertThat(dto.getName()).isEqualTo("Alice");
    verify(repo).findById(1L);
}
Mockito: ArgumentCaptor, Matchers, Spy vs Mock
ArgumentCaptor — captures the argument passed to a mock so you can assert on it:

assertThat(captor.getValue().getTo()).isEqualTo("alice@example.com");

On a Spy, when(spy.method()).thenReturn(x) still calls the real method before stubbing. Use doReturn(x).when(spy).method() to avoid it.

  • Can't mock final classes/methods by default — needs inline mock-maker (add mockito-inline or Mockito 5 default).
  • Static methods: mockStatic(ClassName.class) in Mockito 4+/5 (try-with-resources scope).

WireMock — HTTP stub server. Use when your service calls external REST APIs and you can't/don't want to hit them in tests. 
@SpringBootTest(webEnvironment = RANDOM_PORT)
@AutoConfigureWireMock(port = 0)
class PaymentClientTest {

    @Test
    void chargeReturns200() {
        stubFor(post(urlEqualTo("/charge"))
            .willReturn(okJson("{\"status\":\"ok\"}")));

        // call your service that internally calls /charge
    }
}
Verify outgoing calls: verify(postRequestedFor(urlEqualTo("/charge"))). RestAssured — fluent API testing over HTTP; often used for E2E or full-stack integration tests against a running server: 
given()
    .contentType(JSON)
    .body(new OrderRequest("item", 2))
.when()
    .post("/orders")
.then()

Combine Testcontainers (real DB) + WireMock (external APIs) + RestAssured (HTTP assertions) for comprehensive integration tests that mirror production without real external dependencies.
Test Naming, AAA Pattern, What NOT to Test
AAA pattern: Arrange (set up data and mocks) → Act (call the method) → Assert (verify outcome). Add blank lines between sections for readability. Naming conventions:
  • methodName_stateUnderTest_expectedBehavior — e.g. findUser_whenNotFound_throwsException
  • given_when_then — BDD style, reads as a sentence
  • Use @DisplayName for prose-style names in reports
What NOT to test:
  • Getters and setters (no logic).
  • Framework behavior — don't test that @Transactional rolls back; Spring's own tests cover that.
  • Lombok-generated code.
  • Configuration beans that just wire things together.
  • Private methods directly — test through the public API; if private logic is complex enough to demand a test, it's a design smell (extract a class).
"I test behavior, not implementation. If I can refactor internals without changing tests, the tests are written at the right level."
Code Coverage (Jacoco) & Contract Testing
Jacoco generates line and branch coverage reports. Integrate in Maven/Gradle; add a minimum threshold to the build.
MetricWhat it measures
Line coveragewhich lines were executed

100% line coverage does NOT mean correctness. You can execute every line and still miss edge cases if assertions are weak. Branch coverage is more meaningful. Don't chase the number — cover critical paths and business logic.
Exclude generated code (Lombok, MapStruct, DTOs) from Jacoco reports via <excludes> in plugin config — otherwise the metric is noise.
Contract Testing (brief):
  • Spring Cloud Contract — producer defines contracts (Groovy/YAML DSL), generates stubs for consumers; consumer tests against stubs; producer CI verifies contracts pass. Keeps producer/consumer in sync without full E2E.
  • Pact — consumer-driven; consumer generates pact files, publishes to Pact Broker; producer verifies. Polyglot-friendly.
"Contract tests give me confidence that a service change won't break consumers, without needing a deployed environment to run against."

Pub/Sub — managed async messaging
Fully managed message bus. Topics → subscriptions. Decouples producers from consumers.
  • At-least-once delivery — duplicates are possible. Consumers must be idempotent.
  • Pull subscriptions — consumer calls pull(), processes, then acknowledge(). Better for batch or latency-tolerant workloads.
  • Push subscriptions — Pub/Sub POSTs to your HTTPS endpoint. Better for Cloud Run / serverless (no long-poll thread needed).
  • Ordering keys — messages with the same key delivered in order to a single subscriber. Throughput tradeoff: one shard per key.
  • Dead-letter topics — after N delivery attempts, message forwarded to DLT. Monitor DLT for poisoned messages.
  • Message retention: up to 7 days. Ack deadline: 10 s–600 s (extend with modifyAckDeadline for slow consumers).
Exactly-once delivery is NOT the default. If you need it, use Pub/Sub Lite with ordering or Dataflow dedup — and accept the latency cost.
"We use push subscriptions into Cloud Run — no long-polling threads, auto-scales with message volume, DLT catches poison messages."
IAM — principals, roles, bindings
IAM policy = set of bindings. Each binding: { role, [members] }. Evaluated at resource level; inherited down the hierarchy (org → folder → project → resource).

Principals: Google account, group, service account, allUsers, allAuthenticatedUsers.
Role types:

  • Predefined — e.g. roles/storage.objectViewer, roles/cloudsql.client. Fine-grained, GCP-managed.

Principle of least privilege — grant only what the workload needs, scoped to the narrowest resource possible (e.g. bucket, not project).

Service accounts — identity for a workload. Attach to VM / Cloud Run / GKE pod. Prefer Workload Identity (GKE) over downloading key JSON files — no key rotation, no secret sprawl.

Jenkins — what it is and why it's still everywhere
Jenkins is an open-source automation server (Java-based) that orchestrates build, test, and deploy pipelines via a massive plugin ecosystem (~1800 plugins). It's self-hosted, which means full control but also full responsibility for ops.
  • Plugin-heavy architecture: Git, Maven, Docker, Kubernetes, Slack, JUnit plugins are standard.
  • Master/agent architecture: master schedules, agents (nodes) execute — can scale horizontally.
  • Configuration: historically UI-driven (Freestyle jobs), now Pipeline-as-Code via Jenkinsfile in the repo is the standard.
"We store Jenkinsfiles in the repo alongside the code — pipeline is versioned, reviewable, and lives with the service it deploys."
Blue Ocean is a modern Jenkins UI plugin — better visualization of pipeline stages, parallel steps, and PR builds. Worth mentioning if asked about DX.

Rebase vs Merge — when and why
Merge: joins two branch histories with a merge commit. Preserves full context of when branches diverged.
Rebase: replays your commits on top of the target branch — linear history, no merge commit. Rewrites commit SHAs.
MergeRebase
HistoryNon-linear, merge commitsLinear, clean log
SHA rewriteNoYes
Safe on shared branchesYesNo — never rebase shared/public branches

# Update feature branch without merge commit:
git fetch origin
git rebase origin/main

# Pull with rebase instead of merge commit:
git pull --rebase origin main

git reset, revert, cherry-pick, stash
git reset — moves HEAD (and optionally index/working tree) backward:
  • --soft HEAD~1: undo commit, keep changes staged.
  • --mixed HEAD~1 (default): undo commit, keep changes unstaged.
  • --hard HEAD~1: undo commit, discard changes. Destructive.
git reset --hard on a pushed commit + force push = rewrites public history. Never on main. Use git revert instead.
git revert: creates a new commit that undoes a previous commit. Safe — preserves history: 
git revert abc1234   # new "Revert" commit, sha stays in log
git cherry-pick: apply a specific commit from another branch: 
git cherry-pick abc1234   # e.g. hotfix to both main and release branch
git stash: shelve dirty working tree temporarily: 
git stash push -m "WIP: auth refactor"
git stash pop       # re-apply latest
git stash list      # see all stashes
Stash is not a backup mechanism. If you stash and forget, it silently rots. Prefer short-lived WIP commits on a personal branch.

PR workflow, conflict resolution, code review etiquette
Standard PR workflow:
  1. Branch from main: git checkout -b feature/JIRA-123-user-auth
  2. Small, focused commits. Push early as draft PR for visibility.
  3. Rebase on main before requesting review: git rebase origin/main
  4. At least 1 approval required (2 for main in critical repos).
  5. Squash-or-merge policy on main keeps history clean.

git rebase origin/main

git add src/main/java/.../UserService.java

  • Block only on correctness/security. Style → non-blocking comment.

11. Security

Authentication vs Authorization
Authentication = who you are (identity verification). Authorization = what you're allowed to do (access control). Always in that order — you can't authorize an unknown identity.
  • In Spring Security: AuthenticationManager handles authn; AccessDecisionManager / method security handles authz.
  • 401 Unauthorized = authn failed (not logged in). 403 Forbidden = authz failed (logged in but no permission).
Interviewers swap these terms on purpose. "401 means unauthorized" — technically yes, but it really means unauthenticated. Say that.
"Authentication establishes identity, authorization enforces policy on that identity. Spring Security keeps these as separate concerns — the filter chain handles authn, method-level annotations handle authz."
Spring Security Architecture
Core pipeline: request → FilterChain → SecurityFilterChain → AuthenticationManager → UserDetailsService → SecurityContext.
  • SecurityFilterChain — bean that defines which filters apply and in what order.
  • AuthenticationManager — delegates to AuthenticationProvider(s); default is DaoAuthenticationProvider.
  • UserDetailsService.loadUserByUsername() — loads user from DB; return UserDetails with username, password hash, granted authorities.
  • SecurityContextHolder — thread-local store for the current authenticated principal. Clear it explicitly in async code.
  • Filters run in a fixed order: UsernamePasswordAuthenticationFilter, BasicAuthenticationFilter, BearerTokenAuthenticationFilter, etc.
WebSecurityConfigurerAdapter is removed in Spring Boot 3 / Spring Security 6. Never mention extending it. Use @Bean SecurityFilterChain method instead.
@Bean
SecurityFilterChain api(HttpSecurity http) throws Exception {
    return http
        .csrf(AbstractHttpConfigurer::disable)
        .sessionManagement(s -> s.sessionCreationPolicy(STATELESS))
        .authorizeHttpRequests(a -> a
            .requestMatchers("/auth/**").permitAll()
            .anyRequest().authenticated())
        .addFilterBefore(jwtFilter, UsernamePasswordAuthenticationFilter.class)
        .build();
}
CSRF — what it is and when to disable
Cross-Site Request Forgery: attacker's page makes the authenticated user's browser send a forged state-changing request (POST/DELETE) to your app. Browser automatically attaches session cookie — server can't distinguish it from a legitimate request.
  • CSRF protection works by requiring a token that the attacker's page cannot read (same-origin restriction).
  • Enable when: server-rendered HTML + session cookies (traditional MVC apps, Thymeleaf).
  • Disable when: stateless REST API using JWT in Authorization header — browser's auto-cookie mechanism is the attack surface; JWTs in headers are not sent automatically.
Never disable CSRF on a mixed app (some endpoints use sessions). Scope it per SecurityFilterChain matcher.
"We disable CSRF on the REST API because we use stateless JWT bearer tokens — there's no session cookie for the attacker to exploit."
CORS — configuration in Spring
Cross-Origin Resource Sharing: browsers block JS from reading responses from a different origin (scheme + host + port). CORS is the server-side opt-in via Access-Control-Allow-* headers.
  • Preflight (OPTIONS) request is sent first for non-simple requests; Spring handles it automatically when CORS is configured.
  • Option 1 — annotation: @CrossOrigin(origins = "https://frontend.com") on controller.
  • Option 2 — global via WebMvcConfigurer:
@Bean
WebMvcConfigurer cors() {
    return new WebMvcConfigurer() {
        public void addCorsMappings(CorsRegistry r) {
            r.addMapping("/api/**")
             .allowedOrigins("https://app.example.com")
             .allowedMethods("GET","POST","PUT","DELETE")
             .allowCredentials(true);
        }

JWT — structure, trade-offs, refresh rotation
JSON Web Token: base64url(header).base64url(payload).signature. Signed (JWS) by default, not encrypted — payload is readable by anyone. Encrypt with JWE if claims are sensitive.
  • Header: alg (RS256, HS256), typ.
  • Payload (claims): sub, iss, exp, iat, custom roles/scopes.
  • Signature: server-side secret or private key — tamper detection only.
Trade-offs:
  • Pro: stateless, horizontally scalable, no session store needed.
  • Con: cannot revoke before expiry without a blocklist (defeats statelessness). Mitigation: short TTL (15 min) + refresh token.
Refresh token rotation: each use of a refresh token issues a new refresh token and invalidates the old one. Detect reuse → revoke entire family (breach signal).
HS256 uses a shared secret — all services that verify the token also know the secret and can issue tokens. Prefer RS256 (private key signs, public key verifies) in multi-service setups.

STORY 1 — Followup Q&A (drill these)

Q: Why JWT and not opaque tokens?

A: Stateless validation — each service validates the token locally using the public JWKS keys without a network call to the auth server. With [SWAP IN: 5-6] services all needing to authenticate inbound requests, opaque token introspection would mean every request hitting a central introspection endpoint, which becomes a latency bottleneck and a reliability dependency. JWTs let us scale horizontally without that coupling. The tradeoff is revocation — you give that up, and you have to engineer around it.

Q: How did you handle token revocation given JWTs are stateless?

A: Short-lived tokens — [SWAP IN: 15-minute] expiry — so the blast radius of a leaked token is bounded by time. For genuine emergencies (compromised service account), we had a Keycloak admin-side kill switch that could invalidate the client credentials, which effectively blocked new token issuance. We also maintained a thin deny-list in Redis for explicitly revoked tokens — it's an anti-pattern against stateless JWTs but it gave us an emergency brake for high-value service accounts. The Redis check was gated: only checked if the token's jti appeared in a suspect set, so it didn't add latency on the happy path.

Q: What grant type did you pick and why?

A: Client Credentials grant — this is service-to-service, machine-to-machine. There is no human user in the loop. The airline partner's back-office system authenticates with a client ID and secret, gets a token scoped to what that system is allowed to do. Authorization Code with PKCE is for user-facing flows where you need the user to delegate permissions. Client Credentials is the right grant for B2B API auth.

Q: What is PKCE and did you need it here?

A: PKCE is Proof Key for Code Exchange — it prevents authorization code interception attacks in the Authorization Code flow, specifically for public clients (mobile apps, SPAs) that can't safely store a client secret. You generate a code verifier and send a hashed version (code challenge) with the auth request; when exchanging the code for a token, you prove you have the original verifier. We didn't need it — Client Credentials has no authorization code step, and our clients were confidential server-side systems that could hold a client secret. If we were building a user-facing flow for airline agents in a browser app, PKCE would be mandatory.

Q: How did you protect against token leakage?

A: Several layers. Short expiry (discussed above). HTTPS everywhere — tokens only in transit over TLS, never in query parameters (always Authorization header). We logged token usage by jti and client_id but never the raw token. On the client side, tokens were held in memory and not persisted to disk or logs. We also set the aud claim and validated it strictly — so a token issued for one of our services couldn't be replayed against another.

Q: How did you handle scope vs role — what's the difference?

A: Scopes define what the client application is allowed to do — it's about the API surface. Roles define who the user or service principal is — it's about identity. In OAuth 2.0 resource server context, we worked with scopes: write:profiles means this client is authorized to call write endpoints. Roles (ROLE_ADMIN, ROLE_PARTNER) are an application-internal concept. We mapped Keycloak realm roles to granted authorities via the converter, but the primary guard was scope-based because that's what the spec is designed for in client credentials flows — the client's level of access, not its identity category.

Q: How did you test the security layer?

A: Three levels. Unit tests with @WithMockJwt (from spring-security-test) — set up tokens with specific claims and scopes, assert that @PreAuthorize rules fire correctly. Slice tests with @WebMvcTest — verify 401 on missing token, 403 on wrong scope, 200 on valid token. Integration tests with Keycloak in Testcontainers — actually issued real tokens using the client credentials flow against a containerised Keycloak, called the API, verified the full chain including JWKS fetch and claim extraction. The Testcontainers tests were slower but they caught two bugs the mocked tests missed — one in the audience validation config, one in the converter clock skew handling.

Q: What would you do differently if you started today?

A: Two things. First, I'd use Spring Boot 3's auto-configuration more aggressively — set spring.security.oauth2.resourceserver.jwt.issuer-uri and let Boot discover the JWKS URI from the OIDC discovery endpoint rather than hardcoding the JWKS URI. Less config, auto-rotates when Keycloak moves its key endpoint. Second, I'd push harder for OpenTelemetry trace context propagation through the security filter — we retrofitted trace ID correlation later and it was painful. Starting with it wired into the security chain from day one would have made debugging auth failures in distributed traces much easier.

A — Action:

1. Custom annotation:

@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface TrackJourneyStep {
    String name();
    boolean carrierAware() default true;
    String carrierCodeExpression() default ""; // SpEL against method args
}

2. Spring AOP @Around aspect:

@Aspect
@Component
@RequiredArgsConstructor
public class JourneyMetricsAspect {

    private final MeterRegistry registry;
    private static final Logger log = LoggerFactory.getLogger(JourneyMetricsAspect.class);

    @Around("@annotation(track)")
    public Object recordStep(ProceedingJoinPoint pjp, TrackJourneyStep track) throws Throwable {
        String carrierCode = resolveCarrierCode(pjp, track);
        Timer.Sample sample = Timer.start(registry);
        String outcome = "success";
        try {
            return pjp.proceed();
        } catch (Exception ex) {
            outcome = classifyException(ex);
            registry.counter("journey.step.error",
                "step", track.name(),
                "carrier", carrierCode,
                "reason", outcome
            ).increment();
            throw ex;
        } finally {
            sample.stop(Timer.builder("journey.step.duration")
                .tag("step", track.name())
                .tag("carrier", carrierCode)
                .tag("outcome", outcome)
                .register(registry));
        }
    }

    private String resolveCarrierCode(ProceedingJoinPoint pjp, TrackJourneyStep track) {
        if (!track.carrierAware() || track.carrierCodeExpression().isBlank()) {
            return "unknown";
        }
        // SpEL evaluation against method args omitted for brevity — returns enum-safe string
        return CarrierCodeResolver.resolve(pjp, track.carrierCodeExpression());
    }

    private String classifyException(Exception ex) {
        if (ex instanceof CarrierUnavailableException) return "carrier_unavailable";
        if (ex instanceof ValidationException) return "validation_error";
        if (ex instanceof TimeoutException) return "timeout";
        return "unknown_error";
    }

STORY 2 — Followup Q&A (drill these)

Q: Why not just use Micrometer's @Timed annotation directly?

A: @Timed gives you static tags only — you declare them at compile time on the annotation. We needed runtime values from method arguments: the carrier code from the request object, the booking phase derived from business state. With @Timed you can't say "tag this with the carrier code extracted from the first argument". You'd end up with manual Timer code inside every service method, which is the boilerplate we were trying to avoid. Custom AOP with SpEL argument extraction was the missing piece.

Q: Why not OpenTelemetry — that's the standard now?

A: Fair challenge. At the time we were on Spring Boot 2.x, Micrometer was already wired to Prometheus, and introducing OTel would have meant either a full stack migration or running two parallel pipelines. The ROI wasn't there for a 2-week instrumentation task. Today I'd use the Micrometer Tracing / OTel bridge introduced in Spring Boot 3 — you get OTel-compatible trace propagation while keeping the Prometheus/Grafana stack. We were also building on top of OTel conceptually (trace IDs, span correlation) — we just used Sleuth/MDC rather than the OTel SDK for the actual propagation.

Q: How did you avoid cardinality explosion in Prometheus?

A: Strict rules on what can be a tag value. Carrier codes come from a validated registry enum on startup — unknown carrier codes are bucketed as "unknown", not passed through as-is. Error reasons are a closed enum in the aspect — not the exception class name, not the message, not anything user-influenced. We explicitly banned booking reference numbers, route codes, user IDs, or anything with unbounded cardinality from tags. Every tag set was reviewed before merge. Prometheus with exploded cardinality is a slow-motion disaster — you won't notice until your TSDB is burning memory and scrapes start failing.

Q: What happens if Prometheus is unreachable — do you lose metrics?

A: Micrometer with a Prometheus registry is pull-based — Prometheus scrapes our /actuator/prometheus endpoint. If Prometheus is down, our application doesn't know and doesn't care: metrics accumulate in the in-memory registry normally. When Prometheus recovers and scrapes again, it gets the current state. What you do lose is the time-series data for the period Prometheus was down — there's no buffering of historical scrape data on the app side. For push-based registries (like InfluxDB or Statsd) the failure mode is different — drops happen. Pull-based is more resilient from the application's perspective.

Q: How did the AOP aspect handle exceptions and async code?

A: For exceptions: try/catch/finally in the @Around advice — the finally block always stops the timer (so we get a duration even for failed calls), the catch block increments a separate error counter with a classified reason tag, then rethrows. For async: this was the tricky part. @Async methods run on a different thread — the AOP advice runs on the calling thread, so when pjp.proceed() returns a Future or CompletableFuture, the timer stops immediately before the actual work completes. We had to detect CompletableFuture return types in the aspect and attach a whenComplete callback to stop the timer on the async thread. MDC context loss on async threads was solved with an MDCTaskDecorator on the async executor.

Q: How did you measure the overhead of the aspect itself?

A: JMH microbenchmarks — a method with no annotation, the same method with @TrackJourneyStep and carrierAware=false (no SpEL), and with full SpEL resolution. Overhead was sub-millisecond — around [SWAP IN: 0.05-0.1ms] in steady state after JIT warmup. Given the annotated methods themselves took hundreds of milliseconds (network calls to carrier systems), this was well below noise. For high-volume non-network methods (local validation, in-memory lookups) we used the sampling flag — record 1 in every 10 calls for those endpoints.

Q: How did you correlate a metric with a distributed trace?

A: MDC was the bridge. Spring Cloud Sleuth (at the time) populated MDC with traceId and spanId on every thread. Inside the aspect's exception handler, we read MDC.get("traceId") and included it as an exemplar — a Micrometer feature (available from 1.9) that attaches a trace ID sample to a histogram bucket. In Grafana, you can click a metric spike, see the exemplar trace ID, and jump directly to the Jaeger/Tempo trace for that request. For async threads we had to explicitly copy MDC context using the decorator, otherwise the traceId was gone by the time the async work ran.

Q: If you built this today, what would you do differently?

A: I'd use the Micrometer Tracing API (Spring Boot 3) with the OpenTelemetry bridge instead of Sleuth/MDC manually. That gives you W3C traceparent propagation, OTel-compatible spans, and Micrometer metrics all in one coherent model without dual pipelines. I'd keep the custom @TrackJourneyStep annotation and AOP approach — that's still the right ergonomics for domain-aware tags — but I'd have the aspect create explicit OTel spans via the Tracer API rather than just reading MDC. The result would be richer traces (each booking step as a named span with attributes) rather than just correlating metrics to traces by ID. Essentially: same annotation, same AOP, but OTel-native underneath instead of Micrometer-only.
